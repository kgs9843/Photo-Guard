# PR 루프 (에이전트/사람 공통)

목표는 **짧은 PR + 빠른 반복**입니다. 리뷰 코멘트는 “취향”이 아니라 “재현 가능한 규칙”으로 축적합니다.

## PR 만들기 체크리스트

- 변경 범위를 작게 유지한다 (가능하면 1가지 목적)
- 로컬 기준으로 아래를 통과시킨다
  - lint / typecheck
  - 테스트(있다면)
- PR 본문에 아래를 포함한다
  - **Summary**: 왜 바꿨는지 1~3줄
  - **Test plan**: 무엇을 어떻게 확인했는지
  - **Risk / Rollback**: 위험요소와 되돌리는 방법(필요 시)

## 리뷰 코멘트 처리 루프

- 코멘트를 “규칙/가드레일 후보”로 분류한다
  - **버그/안정성**: 즉시 수정 + 재발 방지(테스트/린트 후보)
  - **일관성/구조**: 문서(`docs/architecture`) 또는 린트 후보
  - **취향**: 팀 규약으로 채택할지 결정(채택 시 문서화)
- 같은 코멘트가 2회 이상 나오면 문서로, 3회 이상 나오면 기계적 가드레일(린트/테스트/CI)로 승격한다
